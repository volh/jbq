* Project Overview

jbq is a jq replacement with a genuinely better query language. Written in
OCaml, with a native simdjson-based JSON input path and an internal lazy
sequence/transducer runtime.

Every existing "better jq" (jaq, gojq, fq, query-json) kept jq's syntax and
competed on runtime. jbq competes on language design *and* now has a native
input path that materially changes the performance profile.

** Repository
=~/dev/sandbox/jbq/=

** Status
Working prototype. 82/82 tests passing. CLI functional.

Current parser/runtime state:
- simdjson is the default JSON input path
- top-level array queries use a streamed =Value.Seq= path when the query shape
  is compatible
- non-streamed inputs fall back to a full native =Value.t= parse
- exact big integers are preserved via =BigInt of Z.t=

** Core Design Rules
1. =.field= — dot is always data (field access)
2. =bare_word= — always a function
3. =|= — always composition (pipe)

* Architecture

#+begin_src
jbq 'items | where .price > 100 | map {name, price}' data.json

  ┌────────────┐    ┌────────┐    ┌─────┐    ┌─────────────┐    ┌────────┐
  │ simdjson   │ -> │ Parser │ -> │ AST │ -> │ Interpreter │ -> │ Output │
  └────────────┘    └────────┘    └─────┘    └─────────────┘    └────────┘
   native JSON      Pratt                    transducer        direct
   parse / stream                            pipelines         printer
#+end_src

** Input Path

There are now two native input modes in =bin/main.ml=:

1. *Top-level array stream*
   - enabled automatically when the query shape is compatible
   - builds a =Value.Seq= from a simdjson top-array iterator
   - avoids full-array materialization

2. *Full native parse*
   - used for other inputs/query shapes
   - constructs =Value.t= directly from simdjson

The old Yojson input path is no longer used in active code.

* Runtime Model

** Values
#+begin_src ocaml
type t =
  | Null
  | Bool of bool
  | Int of int
  | Float of float
  | String of string
  | Array of t list
  | Object of (string * t) list
  | Seq of t Seq.t
  | Xd of t * xd
  | BigInt of Z.t
#+end_src

The important change here is =BigInt of Z.t=. Large integer literals are
preserved exactly instead of degrading to float.

** Numeric Semantics

- exact integer arithmetic for =Int= / =BigInt= combinations on:
  - =+=
  - =-=
  - =*=
  - =%=
  - =/=
- mixed integral/float arithmetic promotes to =Float=
- huge integer literals stay exact
- huge integer equality/ordering stays exact

This is strictly better than the previous jbq behavior and better than jq on
large integer arithmetic.

** Transducer Pipeline

Collection ops (=where=, =map=, =take=, =skip=, =unique=, =flatten=) compose
into a single transducer descriptor (=Xd=). Execution is deferred until the
result is realized.

Aggregation (=sum=, =avg=, =count=) folds directly over the sequence/transducer
without materializing an intermediate array.

* Performance Snapshot

Numbers below are from the current =main= tree on 2026-03-09, comparing jq
1.8.1 with the native simdjson jbq path.

** xlarge.json (100K items)

| Query                          | jq       | jbq      | Result                         |
|--------------------------------+----------+----------+--------------------------------|
| =where .price > 400 | take 5=  | ~234.2ms | ~19.1ms  | jbq ~12.3x faster               |
| =count=                        | ~233.3ms | ~70.3ms  | jbq ~3.3x faster                |
| =map .price | sum=             | ~315.1ms | ~82.8ms  | jbq ~3.8x faster                |
| =.=                            | ~490.5ms | ~493.9ms | near parity, jq still ~1% ahead |

** Interpretation

- Top-level array streaming is a major win for early-exit and single-pass
  reductions.
- Identity is no longer catastrophically slower than jq, but not yet better.
- The remaining work is no longer "replace Yojson"; that is done.

* Parser Details

** Pratt Parser

Expressions are parsed with a Pratt parser. Prefix parsing handles literals,
unary operators, arrays/objects, =if=, =let=, lambdas, and root dot access.
An infix binding-power table handles precedence and associativity in one place.

** Function Arguments and Postfix Binding

Whitespace separates implicit function arguments:
=pick .name .email=

Postfix operators only bind when attached to the expression they extend:
- =.items[0].name= chains
- =.items [0]= does not
- =.name?= works, =.name ?= does not

* JSON Correctness Coverage

The test suite now includes:
- huge signed/unsigned integers
- huge negative integers
- exact bigint arithmetic
- Unicode strings
- escaped keys
- surrogate-pair input
- malformed JSON
- deep nested arrays/objects
- streamed path coverage for supported top-array ops

* Known Limitations

- Arrays and objects are still list-backed in the runtime
- Top-level array streaming is query-shape dependent, not universal
- Identity still slightly trails jq
- Build currently assumes a system simdjson installation
- =split= still only supports single-character separators
- No parenthesized/comma call syntax yet
