* Project Overview

jx is a jq replacement with a genuinely better query language. Written in OCaml.
Not "jq in another language" — a new language for JSON.

Every existing "better jq" (jaq, gojq, fq, query-json) kept jq's syntax and
competed on runtime. jx competes on language design.

** Repository
=~/dev/sandbox/jx/=

** Status
Working prototype. 44/44 tests passing. CLI functional.

** Core Design Rules
1. =.field= — dot is always data (field access)
2. =bare_word= — always a function
3. =|= — always composition (pipe)

** Module Dependency Graph
#+begin_src
  Ast  <-- Error  <-- Lexer  <-- Parser
   ^                                |
   |                                v
  Value <-- Interpreter <-------- (parse_expr for interpolation)
               ^
               | dispatch_ref (mutable)
               v
           Stdlib_fns
               |
           Printer <-- main.ml
#+end_src

The =dispatch_ref= mutable breaks a cyclic dependency between =Interpreter=
and =Stdlib_fns=. Interpreter calls stdlib via the ref; =Stdlib_fns= calls
=Interpreter.eval= directly. One-way dependency at the module level.

* Language Features (Implemented)

** Field Access
#+begin_src bash
jx '.name'                        # top-level field
jx '.user.address.city'           # nested
jx '.[0]'                         # array index
jx '.[1:3]'                       # slice
jx '.[-1]'                        # negative index
jx '.name?'                       # optional (null if missing)
jx '.name'                        # strict (error if missing)
#+end_src

** Pipes
#+begin_src bash
jx '.items | where .price > 100 | map .name'
jx '.name | lower | truncate 20'
#+end_src

** Object Construction with Punning
#+begin_src bash
jx '{name, price}'                # => {name: .name, price: .price}
jx '{name, total: .price * .qty}' # mixed
#+end_src

Punning checks env first (for let-bound variables), then falls back to
field access on input.

** String Interpolation
#+begin_src bash
jx '"hello ${.name}, age ${.age}"'
jx '"total: ${.price * .qty}"'
jx '"literal dollar: \$${.price}"'   # escaped $
#+end_src

Implementation: lexer detects =${= inside strings, collects expression source
until balanced =}= (tracking brace depth), emits =INTERP_STRING= token with
parts. Parser creates sub-parser for each expression part. Interpreter
concatenates results via =value_to_string=.

** Operators
#+begin_src
+  -  *  /  %          # arithmetic
== != < > <= >=        # comparison
&& ||                  # logical
!                      # negation
++                     # string concat
??                     # null coalesce
#+end_src

** Control Flow
#+begin_src bash
jx 'if .price > 100 then "expensive" else "cheap"'
jx 'let total = .price * .qty in {name, total}'
jx 'map (x => {name: x.name})'    # lambda
#+end_src

** Implicit Iteration
When input is an array, collection functions iterate implicitly:
#+begin_src bash
echo '[1,2,3,4,5]' | jx 'where . > 3'   # [4, 5]
# No need for: .[] | select(. > 3) | [.]
#+end_src

* Parser Details

** Precedence (low to high)
1. pipe (=|=)
2. or (=||=)
3. and (=&&=)
4. equality (===, =!==)
5. comparison (=<=, =>=, =<==, =>==)
6. null coalesce (=??=)
7. concat (=++=)
8. additive (=+=, =-=)
9. multiplicative (=*=, =/=, =%=)
10. unary (=!=, unary =-=)
11. postfix (=.field=, =[index]=, =?=)
12. primary (literals, identifiers, parens)

** Function Call Parsing
Bare identifiers are parsed as function calls. Arguments are collected until
a terminator token (=|=, =,=, =)=, =]=, =}=, =;=, =EOF=, =then=, =else=, =in=).

Semicolons separate function arguments: =sort_by .price; .name=

If ==>= follows an identifier, it's parsed as a lambda: =x => x.name=

** String Interpolation Parsing
=INTERP_STRING= tokens contain =Lit= and =Expr= parts. The parser creates a
sub-parser state for each =Expr= part's source string and calls =parse_expr=
on it. This means full expression syntax works inside =${...}=.

* Interpreter Details

** Value Types
#+begin_src ocaml
type t =
  | Null | Bool of bool
  | Int of int | Float of float
  | String of string
  | Array of t list
  | Object of (string * t) list    (* preserves insertion order *)
  | Seq of t Seq.t                 (* lazy, internal only *)
#+end_src

** Truthiness
Falsy: =null=, =false=, =0=, =0.0=, =""=, =[]=. Everything else is truthy.

** Null Handling
Strict by default. =.missing= on an object raises =Key_not_found= with
available keys listed and Levenshtein-based "did you mean?" suggestion.
=.missing?= returns =null=.

** Arithmetic
Mixed int/float promotes to float. Division by zero raises error (not infinity).

** Object Punning with Env
={name, total}= checks let-bound env first, then field access. This means
=let x = 1 in {x}= produces ={x: 1}= not ={x: .x}=.

* Error System

** Error Kinds
=parse_error=, =key_not_found=, =index_out_of_bounds=, =type_mismatch=,
=null_access=, =unknown_function=, =arity_mismatch=, =runtime_error=

** Error Display Format
#+begin_src
error[key_not_found]: key "nme" not found

** Suggestion Engine
Levenshtein distance <= 3 triggers "did you mean?" for both field names and
function names. If no close match, lists all available options.

* Lazy Sequences

** The Problem

jq uses generators — every expression can yield multiple values. Powerful but
confusing. Simple "array in, array out" is easier to understand but wastes
memory: filtering 1M elements to find 5 matches still allocates the full 1M
array plus the filter result.

** The Solution: Lazy In, Lazy Out

Collections flow through pipelines as OCaml =Seq.t= when the source is lazy,
and as =list= when the source is eager. The user never sees the difference —
both print as JSON arrays.

The rule: *if input is =Seq=, output stays =Seq=. If input is =Array=, output
stays =Array=.* Laziness propagates; eagerness is preserved.

#+begin_src ocaml
(* In stdlib_fns.ml *)
let wrap_result input seq =
  if Value.is_seq input then Value.Seq seq
  else Value.Array (List.of_seq seq)
#+end_src

** Which Functions Are Lazy?

| Function   | Lazy? | Notes                                 |
|------------+-------+---------------------------------------|
| =where=     | Yes   | =Seq.filter=                           |
| =map=       | Yes   | =Seq.map=                              |
| =take=      | Yes   | =Seq.take= — this is the key one      |
| =skip=      | Yes   | =Seq.drop=                             |
| =flatten=   | Yes   | =Seq.flat_map=                         |
| =unique=    | Yes   | =Seq.filter= with hashtable tracking   |
| =first=     | Yes   | Peeks first element of =Seq=           |
| =sort_by=   | *No*  | Must realize to sort                   |
| =group_by=  | *No*  | Must realize to group                  |
| =reverse=   | *No*  | Must realize to reverse                |
| =last=      | *No*  | Must realize to find last              |
| =sum/avg=   | *No*  | Aggregation consumes the sequence      |
| =count=     | *No*  | Must consume entire sequence            |
| =range=     | Yes   | 0-arg =range= produces infinite =Seq= |

** How It Works: Pipeline Example

#+begin_src bash
range | where (. % 2 == 0) | map (. * .) | take 3
#+end_src

Step by step:
1. =range= produces =Seq(0, 1, 2, 3, ...)= — nothing computed yet
2. =where= wraps it in =Seq.filter= — still nothing computed
3. =map= wraps that in =Seq.map= — still nothing computed
4. =take 3= wraps in =Seq.take 3= — still nothing computed
5. Printer calls =Value.realize= which calls =List.of_seq=
6. *Now* it pulls 3 values through the chain: 0, 4, 16
7. Only 6 elements of the infinite range were ever examined

** The Initial Problem (What Was Broken)

The first implementation had =Seq.t= in the value type but =implicit_iter=
eagerly converted everything to lists:

#+begin_src ocaml
(* BROKEN: old implicit_iter — kills laziness *)
let implicit_iter input f =
  match input with
  | Value.Array xs -> Value.Array (f xs)
  | Value.Seq s -> Value.Array (f (List.of_seq s))   (* <-- realizes! *)
  | _ -> Error.raise_ ...
#+end_src

=range | take 5= would error with "take requires array, got sequence" because
=take= only accepted =Array=, not =Seq=. Infinite pipelines were impossible.

** The Fix

Replaced =implicit_iter= (which forced everything to lists) with =Seq=-aware
dispatch. Each collection function now works on =Seq.t= directly:

#+begin_src ocaml
(* FIXED: where preserves Seq *)
| "where", [ pred ] ->
  require_collection "where" input loc;
  let s = Value.to_seq input in
  let filtered = Seq.filter (fun item -> ...) s in
  wrap_result input filtered    (* Seq in -> Seq out *)
#+end_src

=wrap_result= checks the original input type: if it was =Seq=, return =Seq=.
If it was =Array=, realize and return =Array=. This means arrays still behave
exactly as before (no regression), while sequences stay lazy.

** Benchmark Results

#+begin_src
--- take 10 from different input sizes ---
  eager 10K array | take 10:    0.9ms    1,167 KB allocated
  eager 100K array | take 10:   24ms    22,098 KB allocated
  eager 1M array | take 10:    256ms   225,069 KB allocated
  lazy  infinite | take 10:   0.02ms       24 KB allocated

--- filter + take 5 ---
  eager 1M array | where | take 5:  285ms  318,822 KB allocated
  lazy  infinite | where | take 5: 0.02ms       27 KB allocated
#+end_src

Eager allocation scales linearly with input size (must parse the entire JSON
array). Lazy is constant: 24 KB regardless of conceptual source size.

** Remaining Limitations

*** Aggregation Allocation
=sum= over lazy sequences still allocates O(n) — each =Value.Int= wrapper
costs words as =Seq.fold_left= traverses the thunks. True O(1) would need a
fused numeric fold that unpacks values during iteration. Currently:

#+begin_src
  range | take 1M | sum:  17ms  234,417 KB allocated
#+end_src

Not terrible, but not the constant-memory ideal.

*** JSON Input Is Eager
=yojson= parses the entire JSON file into memory. Lazy sequences only help
with generated data (=range=) and intermediate pipeline steps. A streaming
JSON parser (SAX-style) would be needed to lazily process large JSON files
from disk. This is a Phase 2+ feature.

*** Inherently Eager Operations
=sort_by=, =group_by=, =reverse=, =last= must see all elements. They call
=List.of_seq= internally, realizing the entire sequence. This is correct
behavior — you cannot sort an infinite sequence. Using these on =range=
without =take= will hang.

* Known Limitations & Cuts

** Parser
- =split= only works on single-character separators (=String.split_on_char=)
- Negative number literals parse as unary minus (works for math, fails in =take -1=)
- No multiline comments

** Interpreter
- =dispatch_ref= mutable is a hack for cyclic dependency
- Cross-type =value_compare= falls back to comparing type name strings
- No tail-call optimization for deep pipelines

** Lazy Sequences (Remaining Gaps)
Lazy sequences are implemented and working. See "Lazy Sequences" section
below for full details. Remaining gaps:
- =sum= over lazy seqs still allocates O(n) due to =Value.t= wrapper per
  element in the =Seq.fold_left=. True O(1) would need fused numeric fold.
- JSON input is still parsed eagerly into memory. Lazy sequences only help
  with generated data (=range=) and pipeline transformations. Streaming JSON
  parser would be needed for true streaming from disk.
- =sort_by=, =group_by=, =reverse=, =last= always realize the full sequence
  (inherently non-lazy operations).

** Stdlib Gaps
Not implemented (mentioned in plan):
- =replace=, =starts_with=, =ends_with=, =contains=
- =flatten= with depth control
- =find= (first match), =some= (any match?), =partition=
- =pick= / =omit= arg parsing is fragile (expects AST field nodes)

** Error Messages
- Location tracking is approximate — positions don't always propagate correctly
- Caret pointer sometimes misaligned for multi-token expressions
- Unknown function with Levenshtein > 3 dumps all function names (noisy)

** CLI
- Reads entire stdin into memory (no streaming)
- No =--null-input= / =-n= flag
- No =--slurp= for JSONL input
- No inline JSON as second positional arg

** Tests
- 44 tests (33 core + 11 lazy sequence tests)
- Lazy tests include a proof-of-laziness test (skips 1M elements, would hang if eager)
- No error case tests beyond strict null
- No edge cases: empty arrays, unicode, large numbers, deeply nested objects
- No property-based testing or fuzzing

** Architecture
- No =.mli= interface files — everything is public
- No barrel module for clean API export

* Comparison with jq

#+begin_src bash
# Filter + pick fields
jq  '.items[] | select(.price > 100) | {name, price}'
jx  'where .price > 100 | map {name, price}'

# String manipulation
jq  '.name | ascii_downcase | ltrimstr("mr. ")'
jx  '.name | lower | trim_start "mr. "'

# Group by (jq returns nested arrays, jx returns object)
jq  'group_by(.type) | map({type: .[0].type, items: .})'
jx  'group_by .type'

# String interpolation
jq  '"hello \(.name), age \(.age)"'
jx  '"hello ${.name}, age ${.age}"'

# Null handling
jq  '.missing'          # null (silent)
jx  '.missing'          # error[key_not_found] with suggestion
jx  '.missing?'         # null (explicit opt-in)

# Keys (jq sorts, jx preserves order)
jq  'keys'              # ["a", "b"] (sorted)
jx  'keys'              # ["b", "a"] (insertion order)
#+end_src
