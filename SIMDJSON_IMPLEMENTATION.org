#+TITLE: simdjson Spike Plan
#+DATE: 2026-03-09

* Goal

- [ ] Build a real native simdjson binding inside =jbq= on branch
  =simdjson-spike=.
- [ ] Make the first useful milestone align with jbq's transducer model:
  top-level array iteration.
- [ ] Keep this as a spike until we prove:
  - the binding compiles cleanly with dune
  - the API is safe enough to evolve
  - it offers a credible path toward streaming pipelines

* Scope

** In Scope For This Spike

- [ ] Add a C++ dune stub linked against simdjson.
- [ ] Expose a tiny OCaml wrapper module.
- [ ] Implement a top-level array stream handle:
  - create from JSON string
  - pull next raw element as a JSON string
  - end-of-stream detection
- [ ] Add tests for:
  - simple top-level arrays
  - nested object/array elements
  - invalid root when not an array

** Explicitly Out Of Scope For This First Milestone

- [ ] General SAX/event runtime redesign.
- [ ] Claiming production-readiness.

* Why This Shape

- [ ] simdjson is strongest as a fast incremental parser.
- [ ] jbq already has a lazy/transducer execution model.
- [ ] The cleanest bridge is a native iterator over top-level array elements,
  not a full parser rewrite on day one.

* Risks

- [ ] Lifetime safety across the OCaml/C++ boundary.
  The parsed document and returned string views must not outlive the backing
  buffer.

- [ ] on-demand API complexity.
  The iterator/document model is fast but stateful and easy to misuse.

- [ ] Build system friction.
  dune C++ stubs and linker flags must work reliably on the target toolchain.

- [ ] False confidence.
  Even if the binding works, it does not prove we should replace the input path.

* Benchmark Snapshot

- [X] Profiled experimental top-array mode on real data.
  For =where .price > 400 | take 5= on =bench/data/xlarge.json=:
  - baseline path: full =yojson parse= + =of_yojson convert=
  - simdjson path: ~8 ms init, no full-tree parse, negligible eval/output

- [X] Benchmarked early-exit pipeline against baseline.
  =where .price > 400 | take 5= on =xlarge.json=:
  - baseline: ~432.1 ms
  - simdjson default path on this branch: ~19.1 ms
  - improvement: ~22.6x over =jbq main=
  - jq: ~234.2 ms, so this branch is ~12.3x faster than jq

- [X] Benchmarked full-array reductions against baseline.
  =count= on =xlarge.json=:
  - baseline: ~431.8 ms
  - simdjson default path on this branch: ~70.3 ms
  - improvement: ~6.1x over =jbq main=
  - jq: ~233.3 ms, so this branch is ~3.3x faster than jq

- [X] Benchmarked =map .price | sum= against baseline.
  - baseline: ~452.3 ms
  - simdjson default path on this branch: ~82.8 ms
  - improvement: ~5.5x over =jbq main=
  - jq: ~315.1 ms, so this branch is ~3.8x faster than jq

 - [X] Benchmarked =identity= against baseline.
   - baseline: ~821.1 ms
   - simdjson default path on this branch: ~493.9 ms
   - jq: ~490.5 ms
   - result: near-parity with jq on identity, still ~1% slower

These results are from the direct-native =Value.t= path. On this branch,
simdjson is now the only active JSON input path in code.
