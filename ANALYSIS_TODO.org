#+TITLE: jx as jq Replacement - TODO Report
#+DATE: 2026-03-09

* Current Assessment

- [ ] Reach true jq-replacement status on both semantics and performance.
  Today, jx is a strong prototype with a better language, but it is not yet a
  full replacement for jq on large-input performance or streaming behavior.

- [ ] Preserve the language advantage while hardening the runtime.
  The language is already a differentiator: readable pipes, strict null
  handling, punning, and named collection operators are easier to teach and
  reason about than jq's generator-heavy model.

- [ ] Keep the performance discussion grounded in current measurements.
  Verified locally on 2026-03-09:
  - xlarge identity: jx 720.8 ms vs jq 481.2 ms (jq 1.50x faster)
  - xlarge early-exit query: jx 397.0 ms vs jq 225.7 ms (jq 1.76x faster)
  - JX_PROFILE=1 identity: ~295 ms yojson parse + ~98 ms of_yojson convert

* Must Fix Before Making Stronger Claims

- [ ] Fix operator associativity in the parser.
  Current parser code makes additive and multiplicative operators
  right-associative. Verified behavior:
  - `10 - 3 - 2` returns `9`, should be `5`
  - `20 / 5 / 2` returns `10`, should be `2`
  This is a correctness issue, not a micro-optimization.

- [ ] Add regression tests for associativity and precedence edge cases.
  Cover `-`, `/`, `%`, chained comparisons, `??`, `++`, and mixed pipe/operator
  expressions so future parser refactors do not silently change semantics.

- [ ] Reconcile docs with implementation on streaming.
  The current CLI reads the entire file into memory and fully parses it before
  evaluation. Until streaming exists, docs should not imply jq-like large-file
  streaming behavior.

* Highest-ROI Performance Work

- [ ] Replace the Yojson intermediate tree with a direct parser into `Value.t`.
  Current flow is: file -> string -> `Yojson.Basic.t` -> `Value.t`.
  This guarantees two full tree walks and substantial allocation before
  evaluation even starts. This is the clearest near-term win.

- [ ] Keep profiling before and after the direct parser change.
  Target outcome:
  - remove the `of_yojson convert` stage entirely
  - reduce total parse-side allocation
  - lower the fixed input floor for all queries

- [ ] Implement top-level array streaming into the transducer pipeline.
  This is the change that can make early-exit queries genuinely competitive or
  faster than jq. Right now `where ... | take 5` still pays full parse cost.

- [ ] Define success metrics for the streaming work.
  For large arrays, queries like:
  - `take 5`
  - `where .price > 400 | take 5`
  - `range-like folds over large inputs`
  should scale with consumed output, not full input size.

* Runtime Representation Work

- [ ] Replace list-backed arrays with a representation optimized for indexing.
  Arrays are currently `t list`, which makes indexing O(n), length checks
  frequent, and memory locality poor compared with arrays or chunked vectors.

- [ ] Replace list-backed objects with a representation optimized for lookup.
  Objects are currently `(string * t) list`, which makes field access O(n).
  A jq-like workload performs large numbers of repeated field lookups.

- [ ] Consider an ordered object representation with a side index.
  Desired properties:
  - preserve insertion order for printing and `keys`
  - offer fast lookup for `.field`
  - avoid paying hash-map overhead on every tiny object if possible

- [ ] Reduce avoidable realization of lazy values.
  `Seq` and `Xd` are often converted to lists or arrays at boundaries. The lazy
  execution model helps, but too many paths still force materialization.

* Stdlib Hotspots To Address

- [ ] Rewrite `group_by` to avoid `existing @ [item]`.
  Appending to the tail on every insert is quadratic within each group. Use
  prepend + reverse, or a mutable growable buffer per bucket.

- [ ] Rewrite `sort_by` to cache keys.
  The current comparator reevaluates the sort expression many times. Use a
  decorate-sort-undecorate approach to evaluate keys once per element.

- [ ] Rewrite `unique` to avoid JSON-string serialization for hashing.
  Current behavior hashes each element by converting it through Yojson and then
  to string. That is expensive and defeats some of the transducer win.

- [ ] Review printer paths for duplicate traversal.
  Pretty-printing first estimates compact width and then prints, which adds an
  extra walk for large values. This is better than the old path, but still not
  minimal.

* Evaluation Model Improvements

- [ ] Preserve the transducer design.
  This is one of the strongest implementation choices in the project. Fused
  `where | map | take | sum` pipelines are the correct direction and should
  remain the core execution model.

- [ ] Push more operations into fused execution where semantically possible.
  Today `sort_by`, `group_by`, `reverse`, `last`, and some printer paths force
  realization. Some of that is inherent, but some boundary materialization can
  still be reduced.

- [ ] Keep evaluation cost separate from input cost in profiling.
  The current profile output is already useful. Keep that discipline so future
  work does not confuse parse wins with evaluator wins.

* Product Positioning

- [ ] Keep marketing the language, not only the benchmark gap.
  The strongest story is still:
  - more readable than jq
  - more regular syntax
  - easier to learn
  - fewer semantic surprises

- [ ] Avoid claiming jq parity before streaming and parser work lands.
  Current status is better phrased as:
  - strong language prototype
  - promising runtime architecture
  - still behind jq at scale

- [ ] Treat correctness and performance as equally important for adoption.
  A jq replacement must be trusted first, then optimized. Parser semantics bugs
  can erase the value of performance work if users cannot rely on results.

* Suggested Execution Order

- [ ] 1. Fix parser associativity bugs and add tests.
- [ ] 2. Implement direct JSON -> `Value.t` parsing.
- [ ] 3. Optimize `group_by`, `sort_by`, and `unique`.
- [ ] 4. Introduce top-level array streaming into transducers.
- [ ] 5. Rework array/object runtime representation for lookup/indexing speed.
- [ ] 6. Re-benchmark against jq after each step on the 10K and 100K datasets.

* Target End State

- [ ] Language remains simpler and more readable than jq.
- [ ] Identity and common map/filter workloads reach near-parity with jq.
- [ ] Early-exit queries beat jq on large top-level arrays through streaming.
- [ ] Large-file behavior matches documentation.
- [ ] Benchmark claims are backed by repeatable measurements and stable semantics.
