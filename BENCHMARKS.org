* Setup

- jq version: 1.8.1 (C)
- jx version: 0.1.0 (OCaml 5.4.0)
- 20 runs per test, 3 warmup runs discarded, median reported
- Memory: peak RSS via =/usr/bin/time -v=
- Ratio = jx/jq (below 1.0 = jx wins)
- Platform: CachyOS Linux 6.19.5-3

* Analysis

** Where jx is faster

Small object operations: field access, arithmetic, nested lookups. The typical CLI use
case — pipe a single object through, extract or transform a field. Also =map=, =unique=,
=take=, =count= at the 1K scale.

** Where jq is faster

Everything at scale. The gap grows with data size: ~1.5x at 1K, ~2-3x at 10K, ~2-3x at
100K. Identity pass-through is 3x slower at scale, which points directly at the
parse/represent/print pipeline rather than the interpreter.

** Memory

jq wins consistently once arrays are involved. At 100K items jx uses ~2x the memory.
Small objects are comparable (~5M each, dominated by runtime overhead).

** Bottleneck diagnosis

The scaling behavior (identity is slowest relative to jq) strongly suggests the cost is
in JSON parsing and value representation, not in the jx interpreter itself:

1. *Yojson parsing* — allocates OCaml values for every JSON node. jq's C parser works
   with a compact tagged-union representation.
2. *Value.t representation* — OCaml variant types carry per-constructor overhead. jq
   packs values into 8-byte NaN-boxed slots.
3. *Printer round-trip* — Yojson pretty-printing allocates intermediate strings.

** Possible optimizations

- Streaming JSON parser (avoid full AST materialization for filtered results)
- Compact value representation (NaN-boxing or flat arena)
- Avoid Yojson intermediate: parse directly into =Value.t=
- Output buffering: write directly to stdout without intermediate string
