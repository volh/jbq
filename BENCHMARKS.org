* Setup

- jq version: 1.8.1 (C)
- jx version: 0.1.0 (OCaml 5.4.0, transducer-fused pipelines)
- query-json version: 0.6.1 (OCaml, installed via npm)
- 20 runs per test, 3 warmup runs discarded, median reported
- Memory: peak RSS via =/usr/bin/time -v=
- Ratio = jx/other (below 1.0 = jx wins)
- Platform: CachyOS Linux 6.19.5-3

* Analysis

** Where jx is faster

Small object operations: field access, arithmetic, nested lookups, conditionals.
jx wins 5/6 vs jq at the single-object scale — the typical CLI use case.

At 1K scale, transducer-fused operations (sum, take, unique, where|take) are
competitive with jq (within 0.94-1.07x). The transducer pipeline eliminates
intermediate allocations for composed operations.

** Where jq is faster

Everything at scale. The gap grows with data size:
- 1K: 1-2.3x (transducer ops near parity)
- 10K: 1.4-3.0x
- 100K: 1.4-3.3x

Identity pass-through is consistently the worst ratio (~3x), confirming the
bottleneck is parse/represent/print, not the interpreter.

** jx vs query-json

At small/medium scale jx is faster or tied on most operations.
At large scale query-json edges ahead on some ops (map, sum, take) due to
its own optimized parser. jx wins sort_by at 100K (*0.87x*) due to lower
allocation overhead in the sort path. Memory-wise query-json uses less memory
on pipeline ops but significantly more on identity/sort (up to 363M vs 219M).

** Bottleneck diagnosis

The scaling behavior strongly confirms the cost is in JSON parsing and value
representation, not in the jx interpreter or transducer pipeline:

1. *Yojson parsing* — allocates OCaml =Yojson.Basic.t= polymorphic variant tree,
   then =of_yojson= walks it again to build =Value.t=. Two full tree allocations
   before any pipeline op fires.

2. *Value.t representation* — OCaml variant types carry per-constructor overhead.
   jq uses compact NaN-boxed 8-byte slots.

3. *Output round-trip* — =to_yojson= builds a third tree, then =pretty_to_string=
   allocates the output string. Four traversals for identity.

4. *No streaming possible* — =Yojson.Basic.from_string= returns a fully materialized
   tree. Even =take 1= on 100K array pays full parse cost. Transducers cannot
   optimize below this floor.

** Possible optimizations

- Custom streaming JSON parser → parse directly into =Value.t=, skip Yojson entirely
- Stream top-level array elements → feed directly to transducer pipeline
- Compact value representation (NaN-boxing or flat arena)
- Direct stdout output without intermediate string allocation
